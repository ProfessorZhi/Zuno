03 / 四条主路径不要混在一起

同步聊天路径

Client -> FastAPI -> ChatService -> Single GeneralAgent -> selected capabilities -> SSE response。

知识查询路径

GeneralAgent 或 Knowledge page -> KnowledgeQueryService -> router -> BM25 / RAG / GraphRAG -> evidence。

异步索引路径

Upload / project change -> RabbitMQ -> Workers -> index_version / community_version atomic publish。

记忆提交路径

post_turn_commit -> Raw Event Log -> Summary Compression -> Structured Extraction -> source_event_ids。

04 / 记忆层设计：上下文不是聊天记录拼接

Production-grade memory extraction / retrieval / consolidation 仍是 Target。当前目标是先把 Raw Event Log、摘要压缩、结构化提取和 trace 边界说清楚。

L0 Working Context

一次模型调用真正看到的 instructions、task、selected memory、knowledge evidence 和 selected ToolCards。

L1 Recent Window

最近消息、tool calls、tool results、interrupt / resume 状态。Sliding Window 只在这里使用。

L2 Task Summary

Summary Compression 产物：目标、约束、决策、TODO、artifact refs、open questions。

L3 Long-term Memory

Structured Extraction 产物：durable facts、preferences、procedures、project memories，必须带 source_event_ids。

L4 External Knowledge

RAG / GraphRAG / files / web evidence。可进入上下文，但不是 Agent Memory 本身。

Raw Event Log 是源头

append-only：messages、model outputs、tool calls/results、interrupts、memory updates。Summary 和 Structured Memory 都是派生层，不能替代 Raw Event Log。

Raw Event Log -> source_event_ids -> Task Summary / Structured Long-term Memory -> ContextTrace -> ModelContextPacket

05 / 能力与工具层：先检索 ToolCard，再注入 schema

ToolCard Registry

记录每个能力的身份、用途、输入输出摘要、权限、side effects、成本、健康状态和 owner module。

identityid、name、aliases、type、owner_module contractdescription、input_schema_summary、output_schema_summary policypermissions、side_effects、cost_hint、latency_hint、health

Capability Selection

根据 task/context 做 exact lookup、Native BM25 over ToolCard text、可选 vector retrieval，然后做 permission / health / cost filter。

task/context -> ToolCard recall -> permission / health / cost -> select schema -> CapabilitySelectionTrace

边界规则

Product-level dynamic capability orchestration 仍是 Target。短期目标是让 GeneralAgent 不再每轮看到全部工具 schema。

06 / RAG / GraphRAG / BM25：一个 Router，多条检索路径

Knowledge Router

router 读取 query intent、project state、index availability、requested method，然后解析成真正执行路径。

basicNative BM25 + dense vector + RRF。
localentity / relation / graph neighborhood / source chunks。
globalcommunity reports / global summaries。
driftglobal primer -> local follow-up -> evidence merge。

Fusion / Evidence

多路召回后先 dedup，再 RRF，必要时 optional rerank，最后构建 citation 和 trace。

Query -> Query Understanding -> Query Variants -> Multi-retriever Recall Native BM25 Dense Vector Graph Local Community Global -> Deduplication -> RRF Fusion -> Optional Rerank -> Evidence Check -> Citation Builder -> Trace
auto 是 router，不是第五种 GraphRAG query method。Native BM25 是本地排序算法；Elasticsearch 只是 optional external adapter。domain_pack_id 仍可能存在于 migration alias、DB compatibility、retirement/history tests，不是 active mainline。

07 / Phase 执行方案：按层落地，不横向摊大饼

Phase 05

Memory Engine

Raw Event Log、Summary Compression、Structured Extraction、source_event_ids、ContextTrace。

Phase 06

Capability / Tool Retrieval

ToolCard Registry、Native BM25 capability search、权限 / 健康 / 成本过滤。

Phase 07

Knowledge Retrieval / Fusion

Basic / Local / Global / Drift、dedup、RRF、optional rerank、citation trace。

Phase 08

LangGraph Runtime

显式 prepare_context -> agent_loop -> post_turn_commit，保留单 GeneralAgent 主线。

Phase 09

Product Boundary / Trace / Eval Closure

Typed API、前端可见状态、trace / eval closure、history archive hygiene。

Execution Contract: Phase 05 关闭后才能开 06；06 关闭后才能开 07；07 关闭后才能开 08；08 关闭后才能开 09。所有阶段在测试和证据证明前都保持 Target。
