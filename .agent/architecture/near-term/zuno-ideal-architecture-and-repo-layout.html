Zuno 目标架构与仓库布局蓝图 V5

这份设计把“运行架构、数据架构、上下文管理和文件夹所有权”统一起来。 目标不是机械创建更多目录，而是让每个概念只有一个代码所有者、每类文件只有一个归宿， 最终形成一个 Single GeneralAgent + Context/Memory + Capability System + RAG/GraphRAG 的本地优先模块化单体。

一段话说明整个项目

Zuno 是一个本地优先的 Agent Workspace：Vue Web 和 Electron 负责产品体验， FastAPI 提供认证、类型化 API 与 SSE，Application Service 负责用例编排； 所有对话只进入一个 GeneralAgent。LangChain 提供模型、消息、工具、中间件和结构化输出抽象， LangGraph 提供有状态循环、Checkpoint、Interrupt、恢复与运行轨迹。 Agent 通过 Capability Registry 按需选择 Knowledge、Action Tool、MCP 和 Skill。 KnowledgeQueryService 将普通 RAG 与 GraphRAG Project 查询统一起来； 文档解析、向量/BM25 索引、图构建与社区报告通过 RabbitMQ Worker 异步完成。 PostgreSQL 保存业务、会话和事件真相，Redis 保存热状态与进度， MinIO 保存源文件和大型 Artifact，Milvus、Elasticsearch、Neo4j 分别提供向量、关键词和图查询索引。 Context Orchestrator 在每次模型调用前按 Token Budget 构造上下文， Post-turn Pipeline 在每轮之后写入原始事件、任务摘要和长期记忆。 系统最终用 Evidence、Citation、Trace 和 Eval 建立可验证闭环。

LangChain 与 LangGraph 的边界

LangChain：能力抽象

  • Chat Model / Message / Tool
  • Agent middleware
  • Structured output
  • Prompt 和 Provider 适配

它负责“模型如何调用能力”，不负责整个产品持久状态。

LangGraph：运行状态与恢复

  • State、Node、Edge、Loop
  • Checkpoint / Resume
  • Interrupt / Human approval
  • Context pre-call 与 Post-turn 节点

它负责“一个 Agent 运行如何持续、暂停、恢复和追踪”。

目标可以是：prepare_context → langchain agent loop → post_turn_commit 的 LangGraph， 而不是再创建一个 DomainQAGraph 与 GeneralAgent 并列。

interfaces/http只解析请求、调用 Application、映射响应
↓
application用例编排、事务边界、权限和 job 创建
↓
agent / context / memoryAgent 运行规则

capabilitiesTool/MCP/Skill 抽象
↓ ports implemented by
infrastructurePostgreSQL、Redis、RabbitMQ、MinIO、Milvus、ES、Neo4j、LLM Provider
↖ bootstrap only wires all concrete implementations
禁止反向依赖：agent 不 import api/services； knowledge 不 import FastAPI；application 不直接写 SQL； infrastructure 不决定 Query Method 或 Agent Policy。

索引链路

1
Ingest：源文件进入 MinIO，PostgreSQL 建 Project/Document/Job 元数据。
2
Parse & Chunk：RabbitMQ 触发 Worker，生成稳定 document/chunk IDs。
3
Text Index：Embedding → Milvus；BM25 → Elasticsearch。
4
Graph Index：实体、关系、claims、evidence backlink → Neo4j。
5
Community：社区检测、摘要和报告，绑定 community_version。
6
Publish：原子切换 index version，旧版本可回滚；Serving indexes 都可重建。

模型调用前

ContextOrchestrator.prepare(step)
1. Load pinned instructions
2. Load thread checkpoint
3. Keep recent coherent window
4. Load latest task summary
5. Retrieve scoped long-term memories
6. Add current knowledge/tool evidence
7. Select relevant tools + skills
8. Allocate token budget
9. Compact or evict low-priority items
10. Build ModelContextPacket
11. Emit ContextTrace

每轮结束后

PostTurnPipeline.commit(turn)
1. Append raw events
2. Save LangGraph checkpoint
3. Update task state
4. Extract memory candidates
5. Dedupe / resolve conflicts
6. Promote approved memories
7. Trigger compaction
8. Save summary with source_event_ids
9. Record cost / latency / selection trace

上下文预算规则

优先级内容预算不足时
P0安全、权限、系统指令、当前用户明确约束永不自动移除
P1最新用户问题、未完成 Tool Call/Result、当前 Task State保留完整交互组
P2相关长期记忆、知识 Evidence、Skill 指导按相关性、来源、置信度裁剪
P3旧消息、大型 Tool Result、低相关说明摘要为 Artifact pointer 或移除

完成后的验收定义

  • GeneralAgent 不 import API Service、DAO 或具体 Store。
  • FastAPI Route 只调用 Application Service。
  • RAG 与 GraphRAG 只存在一套 RetrievalPlanner / Orchestrator / Fusion / Trace。
  • Agent Context、GraphRAGProjectSnapshot、Knowledge Evidence 是三种不同对象。
  • Raw Event Log 是会话与记忆的事实源；Summary 可追溯。
  • 包内不再存在宽泛 services/、core/、utils/ 垃圾桶。
  • 根目录不再存在 tracked domain-packs/、data/、reports/。
  • 生成物和本地数据全部 ignored；公开证据有选择地进入 docs/evidence。
  • 文件夹、代码依赖、文档术语、测试和 Agent Reference 完全一致。
