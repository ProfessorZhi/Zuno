# 核心架构

本文档说明 Zuno 当前的核心运行结构，重点回答三个问题：

- 系统如何组织 Agent、工具和外部能力
- 工作台请求是如何执行的
- 为什么当前版本强调单 Agent 运行时

## 总览

Zuno 当前采用 **单主 Agent + 能力治理层** 的实现方式。

主 Agent 在一次请求中可以使用以下能力：

- 普通工具
- MCP 服务
- 知识库检索
- Skill 资产
- 终端与本地文件能力

这些能力在运行时都会被整理成主 Agent 可感知、可调用的能力集合，而不是再拆成多个子 Agent 互相套娃。

## 运行时主链

当前真正对外生效的主链有两条：

- `WorkSpaceSimpleAgent`
- `GeneralAgent`

两条主链都遵循同一原则：

- MCP 作为工具接入
- 知识库作为检索能力接入
- Skill 作为可按需加载的技能资产接入
- 终端能力作为受权限约束的执行能力接入

## 能力分层

为了让主 Agent 更稳定地选择能力，Zuno 将能力分成几类：

### 1. 工具

普通工具包括系统工具、用户工具以及封装后的能力入口。它们适合：

- 数据处理
- 文件转换
- 简单外部调用

### 2. MCP

MCP 用于连接外部服务，例如：

- 飞书
- 地图服务
- 搜索服务

在当前架构里，MCP 不再作为子 Agent 运行，而是直接暴露给主 Agent 使用。

### 3. 知识库

知识库能力用于文档检索与上下文补充。主 Agent 可以在需要时调用检索能力，再基于检索结果生成回答。

### 4. Skill

Skill 是可复用的方法与执行模板。当前实现遵循两条规则：

- 每轮只暴露 Skill 元数据给主 Agent
- 真正命中某个 Skill 时，再按需加载其正文内容

这使 Skill 更像“技能资产”，而不是“临时拉起的子 Agent”。

### 5. 终端

终端能力独立于普通工具，因为它涉及：

- 本地文件系统访问
- 命令执行
- 更高的副作用风险

因此终端能力通常伴随更明确的权限边界与模式切换。

## 工作台执行方式

工作台当前支持两类核心使用方式：

- 聊天模式
- Agent 模式

在 Agent 模式下，系统会把当前会话里启用的 MCP、工具、Skill、知识库等能力整理给主 Agent。用户也可以通过显式输入方式提高命中率，例如：

- `/技能名`

这类显式入口的目标不是引入多 Agent，而是让单 Agent 更稳定地命中正确能力。

## 为什么强调单 Agent

当前阶段的重点不是构建复杂多 Agent 协作，而是先把单 Agent 的能力治理做好：

- 让 Agent 知道自己当前有哪些能力
- 让显式命令稳定命中 Skill、MCP、知识库与终端
- 让用户能在界面里看见、配置、测试这些能力
- 让 Web 与桌面端行为尽量一致

等这层稳定后，再讨论多 Agent 编排才有意义。
