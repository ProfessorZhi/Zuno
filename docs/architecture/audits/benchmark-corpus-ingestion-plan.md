# Benchmark Corpus Ingestion Plan

Program: Benchmark Corpus Ingestion V1

Scope: define how benchmark corpus should map into temporary Zuno knowledge
assets for future real runtime multihop evaluation. This document does not claim
real GraphRAG quality and does not require runtime architecture changes.

## Short Answer

Recommended mapping:

- one dataset -> one temporary eval-only knowledge
- one corpus document row -> one source document/file-equivalent record before chunking
- keep `question_id`, `doc_id`, and `is_gold` in metadata

This keeps benchmark structure auditable without polluting normal user knowledge
spaces.

## 1. How Benchmark Corpus Maps To Zuno Knowledge / Files / Chunks

Mapping layers:

1. corpus dataset file
2. temporary eval-only knowledge
3. source document rows derived from corpus
4. text chunks generated by normal chunking/indexing
5. retrieval/index metadata preserved from source-document metadata

Recommended source-document payload fields:

- title
- text
- dataset
- question_id
- doc_id
- is_gold
- gold_sent_ids
- source=`multihop_eval`

## 2. Dataset Per Knowledge Or Question Per Knowledge

Recommended: one dataset per temporary knowledge.

Do not create one knowledge per question. That would:

- explode knowledge count
- distort retrieval scope
- make cleanup harder
- move evaluation away from realistic multi-document retrieval conditions

## 3. Recommended Operational Shape

Recommended operational plan:

- `hotpotqa_sample50` -> one temporary knowledge
- `2wikimultihopqa_sample50` -> one temporary knowledge
- `musique_sample50` -> one temporary knowledge

Within each temporary knowledge:

- each corpus row becomes one source document
- metadata keeps `question_id`, `doc_id`, `is_gold`

## 4. How To Avoid Benchmark Leakage Into Normal User Knowledge

Use all of these controls:

- dedicated eval-only naming convention
- dedicated ingestion script under `tools/evals/zuno/multihop_eval/ingestion/`
- do not surface these knowledges in normal product demos
- keep source corpus under gitignored `data/evals/multihop/`
- store explicit metadata `source=multihop_eval`

## 5. How To Mark Eval-only Knowledge

Minimum recommended markers:

- knowledge name prefix: `eval_multihop_`
- metadata tag: `source=multihop_eval`
- metadata tag: `eval_only=true`
- optional description suffix: `temporary benchmark ingestion`

## 6. How To Clean Temporary Knowledge

Cleanup should be explicit and scriptable:

1. list temporary knowledges by prefix/tag
2. remove associated source files/documents
3. remove associated text/vector/BM25/graph artifacts
4. verify no eval-only entries remain before re-running a fresh benchmark ingest

Do not depend on manual UI-only cleanup for repeatable benchmark work.

## 7. Recommended Index Build Order

Recommended order:

1. text chunk build
2. embedding build
3. BM25 build
4. graph entity/relation build
5. optional community detection/report

Community detection/report stays optional because this phase is corpus-ingestion
planning, not full GraphRAG benchmark execution.

## 8. Which Stages Call Real Models

Expected model usage in future real runtime eval:

- embedding: `text-embedding-v4`
- rerank: `gte-rerank-v2`
- answer/eval optional: `deepseek-v4-flash`

This phase itself does not need to call those models. It only defines the
ingestion boundary and stackless smoke path.

## 9. Preconditions For Real Runtime Eval

Before Real Runtime Multihop Eval V1:

- benchmark corpus builder must be stable
- stackless ingestion smoke must pass
- temporary knowledge ingestion path must exist
- eval-only cleanup path must be documented or scripted
- model defaults must remain clean
- benchmark data must remain out of Git
- runtime evaluator must still clearly separate mocked, stackless, and real
  GraphRAG results
