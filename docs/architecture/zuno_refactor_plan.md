# Zuno 改造总方案

## 这份文档回答什么

这份文档只回答一件事：

```text
为什么 Zuno 要按现在这套顺序推进，
以及 Phase 1-7 最终共同指向什么目标。
```

它不是逐阶段执行细则。
执行顺序、退出条件、最小测试，以 `plans/zuno-refactor-execution-plan.md` 为准。

## 当前仓库的真实判断

截至当前仓库状态，新的串行账本应按下面理解：

- `Phase 1`：已完成
- `Phase 2`：已完成
- `Phase 3`：已完成
- `Phase 4`：当前串行阶段
- `Phase 5-7`：仍待完成

这意味着现在最重要的不是继续提前声称更后面的能力已经关闭，而是：

1. 保持 `Phase 1-2` 结果稳定
2. 保持文档与展示面主路径稳定
3. 再继续进入分层边界强化和后续技术主线深化

## 改造目标

Zuno 的目标不是停留在“能聊天的本地工具”，而是形成一个：

```text
本地优先、结构清晰、可扩展、可评测的 Agent 工作台
```

它的核心主线应稳定收敛到：

- `LangGraph` 作为主运行时
- `RAG / GraphRAG / BM25` 作为检索主线
- `Domain Pack` 作为领域能力扩展主线
- `Eval / Trace / Citation / Cost Control` 作为验证与治理主线

## 为什么要按现在这套顺序推进

这次改造按新的 `Phase 1-7` 串行推进，不是为了把名字分得更细，而是为了避免不同类型的问题混在一起：

1. 先收运行时入口，否则后面所有结构和文档都没有稳定对象
2. 再收项目结构，否则公开展示面会继续被历史痕迹污染
3. 再收文档和展示面，否则后续技术深化会一直被旧叙事拖住
4. 再强化分层边界，否则能力继续增长时会越做越乱
5. 再深化 `LangGraph + GraphRAG` 主线，否则项目价值无法真正落到代码主链路
6. 再固化评测与证据链，否则“为什么这样做更好”无法稳定证明
7. 最后做面试前总收口，把代码、目录、文档、验证、展示统一成一个版本

## Phase 1-7 的角色分工

### Phase 1：运行时收口与可运行恢复

目标：

- `zuno` 成为主运行时入口
- 高价值 bridge / alias / contract 收口
- 主启动链路可运行

### Phase 2：项目文件夹与结构硬治理

目标：

- 根目录和 `src/` 语义清楚
- `apps / docs / infra / src / tests / tools` 职责明确
- 公开面、私有面、本地面边界清楚

### Phase 3：文档与展示面硬收口

目标：

- 形成稳定公开阅读路径
- README、架构文档、开发文档、phase 账本口径一致
- 过时或冲突叙事不再干扰主阅读路径

### Phase 4：分层架构与运行时边界强化

目标：

- 控制层、服务层、DAO、基础设施边界更硬
- 高价值运行时路径减少跨层缠绕

### Phase 5：LangGraph + GraphRAG 主线深化

目标：

- `LangGraph` 真正成为主运行时骨架
- `GraphRAG` 动态更新和两档检索主线落地
- `Domain Pack` 主链路价值更明确

### Phase 6：评测与证据链固化

目标：

- 本地 embedding、本地评测、compare matrix、自动化验证稳定可复现
- 展示证据和自动化结果一致

### Phase 7：面试前总收口

目标：

- 代码、目录、文档、评测、展示面、讲解口径最终一致
- 仓库达到正式 GitHub 展示水位

## 面试前最终要达到什么

最终状态不应是“做了很多功能”，而应是下面这些事实同时成立：

1. 项目结构清晰
2. 分层边界清晰
3. `LangGraph + GraphRAG` 是真实主线，不是表层包装
4. 普通 / 增强两档检索可解释
5. `GraphRAG` 动态更新有稳定落点
6. 本地 embedding 与本地评测能提供证据
7. 自动化测试、文档口径、GitHub 展示面相互一致

## 文档同步规则

每完成一个大阶段，都必须回看这些入口是否仍然对齐：

- `README.md`
- `docs/README.md`
- `docs/architecture/README.md`
- `docs/architecture/plans/README.md`
- `docs/architecture/specs/README.md`

原则很简单：

```text
已经解决的问题，不再继续挂在主阅读路径里；
仍未解决的问题，留在后续 phase，而不是提前宣称关闭。
```
